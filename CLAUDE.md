# ТЗ для Claude Code — Education Process Platform (EPP)

## Контекст

Это дипломная работа по направлению 09.03.04 «Программная инженерия». Система — **не LMS**. Это веб-платформа **управления и мониторинга образовательного процесса** для учреждений ДПО и ДО (АНО ДПО «ИнБИТ»). Защита послезавтра. Приоритет — рабочий, чистый функционал по списку ниже. Дизайн будем делать отдельно.

## Стек

- **Backend:** Python 3.10, FastAPI 0.110, SQLModel ORM, PostgreSQL 18, Uvicorn, JWT (HS256), bcrypt
- **Frontend:** React 19, TypeScript 5.4, Vite 5.2, Tailwind CSS 4.0, React Router 6.22
- **Инфра:** Docker Compose

База создана на основе открытого шаблона `tiangolo/fullstack-fastapi-template` (MIT). Вся предметная логика — кастомная.

## Таблицы БД (текущие)

`user`, `program`, `module`, `group`, `enrollment`, `progress`, `lesson`, `admissionrequest`, `achievement`, `userachievement`, `userpoints`

## Роли

- **admin** — полный доступ
- **teacher** — доступ к своим группам и программам
- **student** — только свои данные

## Задача 2 — Целевой функционал (что должно работать)

### АДМИНИСТРАТОР

#### Мониторинг (дашборд)
- Сводная статистика: количество активных программ, групп, обучающихся, необработанных заявок
- Список всех групп с процентом прохождения (агрегат по `progress`)
- Список обучающихся с отставанием (прогресс < 30% при прошедшем > 50% срока)
- Топ обучающихся по прогрессу

#### Управление пользователями
- CRUD пользователей: создать (с временным паролем), редактировать (ФИО, email, роль), деактивировать (без удаления — поле `is_active = false`)
- Фильтр по роли, поиск по ФИО/email

#### Приём заявок (`admissionrequest`)
- Список заявок с фильтром по статусу (new / approved / rejected)
- Одобрить заявку → автоматически создать учётную запись (если email новый) → зачислить в выбранную группу → сменить статус на `approved`
- Отклонить → статус `rejected`

#### Экспорт данных
- Экспорт списка обучающихся группы с прогрессом → CSV
- Экспорт всех заявок → CSV
- Если уже есть Excel (XLSX) — оставить, если нет — только CSV достаточно

---

### ПРЕПОДАВАТЕЛЬ

#### Свои группы
- Список групп, где teacher_id = текущий пользователь
- Внутри группы: список обучающихся, прогресс по модулям (таблица: строки — студенты, столбцы — модули)

#### Отметка прогресса и посещаемости
- Изменить статус прогресса студента по модулю: `not_started` / `in_progress` / `completed`
- Выставить оценку (числовую или текстовую — что есть в схеме)
- Отметить посещаемость занятия: если в таблице `lesson` нет поля `attendance` — добавить простую связь или JSON-поле `attended_user_ids`

#### Календарь занятий
- CRUD занятий своих групп: название, дата/время, продолжительность (минуты)
- Отображение списком, отсортированным по дате (календарный вид опционально)

#### Создание и редактирование программ
- Преподаватель может создать программу (статус `draft` или сразу `active` — без workflow согласования)
- Добавить/редактировать/удалить модули программы
- **Нет** workflow «отправить на согласование» — это вырезаем

#### Индивидуальная траектория обучающегося
- На странице студента внутри группы: текстовое поле «Рекомендации / индивидуальный план» (поле `individual_notes` в таблице `enrollment` или отдельная таблица)
- Преподаватель может выбрать, какие модули программы обязательны для конкретного студента (если слишком сложно — только текстовое поле, этого достаточно)

---

### ОБУЧАЮЩИЙСЯ

#### Личный кабинет
- Список программ, в которых зачислен, с процентом прохождения
- Рекомендации от преподавателя (текст из `individual_notes`)

#### Каталог программ
- Публичный список активных программ ДО и ДПО (без авторизации или с авторизацией — на усмотрение)
- Карточка программы: название, описание, объём часов, список модулей

#### Календарь занятий
- Список занятий своих групп, отсортированный по дате

#### Достижения
- Список полученных достижений и накопленных очков
- Достижения начисляются автоматически при завершении модуля / программы (если логика уже есть — оставить, если нет — добавить минимальную: завершил первый модуль → достижение «Первый шаг»)

## Задача 4 — Полировка

1. Все формы должны показывать inline-ошибки валидации (не только alert/toast)
2. Загрузки должны показывать spinner / skeleton
3. После успешного действия (создать, сохранить, одобрить) — toast-уведомление
4. Пустые состояния: если списков нет — показывать текст «Нет данных» или иллюстрацию, не пустой экран
5. Все таблицы должны быть отсортированы по умолчанию (группы по дате, студенты по ФИО, заявки по дате)
6. На мобильных (< 768px) система не обязана быть идеальной, но не должна быть сломана

После всех изменений:

1. Запусти `docker compose up --build` и убедись, что всё поднимается без ошибок
2. Пройди вручную (или через тесты) сквозной сценарий:

3. Убедись, что `/docs` (Swagger) открывается и все эндпоинты задокументированы
4. Проверь, что `docker compose down && docker compose up` не теряет данные (volume для PostgreSQL примонтирован)
